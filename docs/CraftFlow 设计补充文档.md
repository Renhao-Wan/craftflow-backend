# CraftFlow 设计补充文档

## 它解决了什么致命纠结

**1. “是不是和直接调豆包没区别？” → 不，现在有本质区别。**
* **渐进式创作流 (Map-Reduce)**：豆包一次生成长文会乱，而你通过“大纲→并发写章节→组装”的模式，用工程手段解决了上下文长度限制。这是大模型做不了的结构化控制。
* **多阶润色流 (动态路由 + 对抗循环)**：豆包只能你一句我一句地改。你设计了“极速/对抗/事实核查”三档弹性算力，尤其是三档里自动调用 **Search Tool/REPL 沙箱**去验证事实，这是豆包绝无可能主动完成的多步自动化验证。
* **强制 HITL + 状态持久化**：豆包关闭对话窗口，历史就靠聊天记录。而你的图可以在“大纲确认”和“定稿/转润色”两个关键决策点**自动挂起**，并将完整状态存入数据库，允许用户随时回来恢复。这不是聊天，这是一个**可中断的长周期有状态任务**。

**2. “是不是硬塞的技术栈？” → 这次很自然。**
* **`Send` API 并发**：是为了解决长文章节并发生成的真实业务需求。
* **`Checkpointer`**：是为了支持大纲确认、断点续传的刚需，不是摆设。
* **动态路由 (Strategy Pattern)**：是因为业务真有“快速格式化”和“深度核查”的不同 SLA 要求，模式落地得非常巧妙。
* **异步任务 + 轮询**：是因为 AI 推理确实慢，用这模式解决 HTTP 超时是天经地义，更体现了设计 Java 对接时的架构能力。

**3. “是不是又成了烂大街的写作工具？” → 不会，因为你的叙事变了。**
你的叙事不是“我做了个 AI 写作助手”，而是 **“我用 Map-Reduce 架构解决了长文生成质量失控问题，并用条件路由实现了分级润色与多智能体核查”** 。这是**架构师视角**，不是调包侠视角。

## 问题

### 问题一：中断“展示草稿等待审阅”是否有必要？

确实属于过度设计。它的逻辑完全可以由前端来承载：

1. **前端收到最终 `draft`**（无论是流式还是轮询结果）。
2. **前端展示给用户**，用户阅读后，前端界面上有两个按钮：“完成” 和 “进一步润色”。
3. **用户点击“进一步润色”** → 前端直接调用 `/api/v1/polishing`，把刚刚拿到的 `draft` 作为 `content` 参数传入。
4. **用户点击“完成”** → 结束。

**为什么这样更好？**
* **图的状态更干净**：Creation Graph 的职责就是“生成文章”，生成完就结束。不要让“润色”的逻辑污染创作图。
* **解耦**：润色接口变成了一个独立能力，不仅可以在生成后调用，也可以被 KiwiHub 或其他服务单独调用（比如用户手动粘贴一篇文章来润色）。
* **总结**：“创作与润色两个状态机完全解耦，通过 API 层面的组合而非图内嵌套实现业务流程串联，体现了高内聚低耦合的架构思想。”

### 问题二：用户输入阶段，是否需要补充信息的中断循环？

**对于这个项目，没必要。它不是核心亮点，反而会拖进体验很差的泥潭。**

理由如下：

1. **AI 不是搜索引擎**：大模型根据已有信息进行推理和创作。如果系统不知道用户想要什么风格、什么长度、什么受众，它应该直接生成一个“默认合理”的版本，或者在前端用一个简单的表单来问，而不是在 LangGraph 的状态机里设计一个复杂的“追问 - 回答 - 再追问”循环。**用前端表单解决信息收集，比用状态机追问要简单十倍，体验也好十倍。**
2. **核心展示点不是这个**：核心叙事是“Map-Reduce 解决长文失控”、“多阶润色与动态路由”、“Checkpointer 实现有状态长周期任务”。信息补全循环只是一个常规的前端交互逻辑，放在 LangGraph 里显得大材小用，而且会让面试官的注意力从你真正的亮点上转移。
3. **如果真的要做，复杂度会失控**：需要定义“什么算信息缺失”、设计追问策略、处理用户在追问中突然想改主意的情况、管理多轮追问状态……这个方向的投入产出比极低。

**替代方案（如果面试被问到）：**

> “我在创作图的 `PlannerNode` 里设计了一个可选的 `missing_info` 字段。如果检测到用户输入过于笼统，会先在 `interrupt` 暂停，让前端一次性收集所有补充信息（风格、长度、受众），然后注入后继续。但我认为对于核心流程来说，前端表单收集信息是更成熟的交互方案，避免了将简单的信息收集复杂化为与 Agent 的多轮博弈。”

## 实事求是：几个需要警惕的陷阱

没有完美的设计。在真正写代码前，请务必意识到这几点：

1. **“多阶润色流”里的 Mode 2 对抗循环可能很耗 Token，且效果未必比单次强 Prompt 好**。这不是你的技术问题，而是目前 LLM 的自我批评能力有限。面试时如果被问到，你的准备是：“我设计了 mode 1/2/3 的弹性机制，2 是实验性的，通过迭代和主编批判来尝试突破单次推理上限，并设置了 3 次封顶防止无限循环。”
2. **Map-Reduce 的“整体上下文”是难点**：章节并发生成后，是否真的能丝滑衔接？需要让每个 `WriterNode` 在 Prompt 中都拿到大纲整体和相邻章节摘要，否则会是生硬的拼接。
3. **开发复杂度**：LangGraph 的 `Send` API、`interrupt` 与前端配合、PostgreSQL 状态持久化，这些组合起来的学习曲线比较陡峭。
